# readThings - Development Plan

> A mobile app (Android + iOS) that uses the camera to detect real-world objects,
> shows their names in multiple languages with pronunciation, speaks the word aloud,
> and saves items to a personal vocabulary library.

---

## Architecture Decision: Hybrid ML Pipeline

### Why Hybrid?
We combine two on-device ML systems to get the best of both worlds:

1. **ML Kit Image Labeling** (fast detection, ~30ms/frame)
   - Real-time camera label overlay
   - 400+ object categories
   - Tiny model (~5-15MB)

2. **Gemma 4 E2B** (rich translation, ~1-3s/query)
   - Replaces bundled dictionaries AND cloud translation API
   - 140+ languages with pronunciation/phonetics in one call
   - Open-ended vocabulary (no fixed dictionary limit)
   - ~1.3GB model (GGUF Q4_K_M)

### Data Flow
```
Camera Frame (30fps)
    → ML Kit frame processor (~30ms)
    → English label: "coffee cup" (0.92)
    → Gemma 4 E2B (~1-3s)
    → {
        "ja": { "word": "コーヒーカップ", "phonetic": "koohii kappu" },
        "vi": { "word": "cốc cà phê", "phonetic": "kok ka fe" },
        "ko": { "word": "커피잔", "phonetic": "keopi jan" }
      }
    → Display + Cache in SQLite
```

## Database Schema

```sql
-- Saved vocabulary items
CREATE TABLE library_items (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  image_path TEXT NOT NULL,
  created_at INTEGER NOT NULL,          -- unix timestamp
  is_favorite INTEGER DEFAULT 0
);

-- ML Kit detection results per item
CREATE TABLE item_labels (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  item_id INTEGER NOT NULL REFERENCES library_items(id) ON DELETE CASCADE,
  english_label TEXT NOT NULL,
  confidence REAL NOT NULL
);

-- Gemma translation cache (never translate same word twice)
CREATE TABLE translation_cache (
  english_label TEXT NOT NULL,
  target_language TEXT NOT NULL,
  translated_word TEXT NOT NULL,
  phonetic TEXT,
  PRIMARY KEY (english_label, target_language)
);
```

## Implementation Phases

### Phase 1: Project Scaffold + Camera Preview
- Initialize Expo project with `create-expo-app`
- Install and configure `react-native-vision-camera` with Expo config plugin
- Set up tab navigation: Camera | Library | Settings
- Camera preview full-screen with capture button
- Photo capture → save to documentDirectory
- Platform permissions (camera) on both Android and iOS

**Exit criteria:** App opens to camera, user can take photo, navigation works.

### Phase 2: ML Kit Real-Time Detection
- Write native frame processor plugin (Swift ~50 lines, Kotlin ~50 lines)
- ML Kit Image Labeling on each camera frame (throttled to 1 per 500ms)
- DetectionOverlay component: floating labels with confidence
- Capture mode: freeze frame → run detection → show result screen
- Detection store (Zustand) for current labels

**Exit criteria:** Point at object, see English labels in real-time. Capture shows labels.

### Phase 3: Gemma 4 Translation Engine
- Integrate `llama.rn` with Gemma 4 E2B GGUF model
- Model download flow: WiFi check, progress bar, persistent cache
- Prompt engineering: English label → JSON with translations + phonetics
- Translation cache in SQLite (Drizzle)
- Translation hook: check cache → call Gemma → cache result
- Display translations on capture result screen

**Exit criteria:** Capture photo of "cat" → see translations in selected languages with phonetics.

### Phase 4: TTS + Save to Library
- `expo-speech` wrapper: speak word in target language
- Speaker button on each TranslationCard
- Save flow: capture result → save to library_items + item_labels
- Library grid screen with photo thumbnails + top label
- Library detail screen: full photo + all translations + TTS per word

**Exit criteria:** Full capture → translate → hear → save → browse flow works.

### Phase 5: Settings + Polish
- Language picker (flags + names, multi-select for target languages)
- TTS speech rate slider
- Model management: download/delete Gemma, show storage used
- Onboarding (3 screens on first launch)
- Error states: no camera permission, no model downloaded, TTS unavailable
- Dark/light theme

**Exit criteria:** Complete app, ready for internal testing.

### Phase 6: Release
- EAS Build production profiles
- Android: AAB for Google Play
- iOS: IPA for App Store / TestFlight
- App icons, splash screen
- Privacy policy (camera usage, on-device processing, no cloud data)

## Gemma 4 Prompt Strategy

### Translation Prompt Template
```
You are a translation assistant. Given an English word or phrase for a physical object,
provide translations with pronunciation guides.

Input: "{english_label}"
Target languages: {language_list}

Respond in JSON only, no other text:
{
  "translations": [
    {
      "lang": "ja",
      "word": "translated word in native script",
      "phonetic": "romanized pronunciation"
    }
  ]
}
```

### Prompt Notes
- Keep system prompt minimal to reduce latency
- Request JSON output for reliable parsing
- Include fallback: if Gemma returns malformed JSON, show English label
- Cache every successful translation → subsequent lookups are instant (0ms from SQLite)
- Batch labels when possible: translate 3-5 labels in one prompt to amortize inference cost

## Key Concerns and Risks

### HIGH RISK

#### 1. CameraImage → ML Kit Format Conversion
- **Problem:** Vision Camera v4 frame processors pass native buffer formats that differ between platforms (CMSampleBuffer on iOS, ImageProxy on Android). Converting these to ML Kit's InputImage is the trickiest integration.
- **Mitigation:** Follow Vision Camera's official frame processor plugin guide exactly. Reference existing plugins (face detection, barcode) as templates. Budget extra development time for this step.
- **Note:** This is the single highest-risk item in the entire project.

#### 2. Gemma Model Size (1.3GB download)
- **Problem:** Users must download a 1.3GB model before translations work. Poor UX if not handled well. Storage-constrained devices may not have space.
- **Mitigation:** WiFi-only download by default. Resume-capable downloads. Clear progress UI. Storage check before download. Provide a "lite mode" that falls back to basic English-only labels if model not downloaded.

### MEDIUM RISK

#### 3. Gemma Response Quality / Parsing
- **Problem:** LLM output is non-deterministic. JSON may be malformed. Translations may be inaccurate for rare objects. Phonetics may be inconsistent.
- **Mitigation:** Strict JSON parsing with fallback. Cache known-good translations. Allow user to flag/correct bad translations (future feature). Temperature = 0 for deterministic output.

#### 4. No Expo Go (dev client required from day 1)
- **Problem:** Both Vision Camera and llama.rn require native modules. Cannot use Expo Go for development.
- **Mitigation:** Set up dev client builds early. Use `npx expo run:ios` / `npx expo run:android` for local dev. EAS Build for team distribution.

#### 5. Battery Drain from LLM Inference
- **Problem:** Running Gemma 4 E2B on-device is compute-intensive. Extended use will drain battery.
- **Mitigation:** Only run Gemma when user captures (not on live preview). Aggressive caching means repeated objects are free. Could add a "battery saver" mode that skips translation.

#### 6. ML Kit Label Quality
- **Problem:** ML Kit may return generic labels ("food" vs "sushi", "animal" vs "cat") depending on confidence and model version.
- **Mitigation:** Use lower confidence threshold (0.4) to get more specific labels. Show top 5 labels and let user pick the most relevant. Consider custom TFLite model in future.

### LOW RISK

#### 7. TTS Language Availability
- **Problem:** Some languages may not be installed on the user's device.
- **Mitigation:** Check `expo-speech` language availability. Show "Download voice in device settings" prompt if unavailable.

#### 8. Device Compatibility (Gemma)
- **Problem:** Gemma 4 E2B needs ~2-3GB RAM. Older/budget devices may struggle.
- **Mitigation:** Check available memory before loading model. Show device requirements on model download screen. The app still works without Gemma (English labels only).

## Notes

- **Why not flutter_gemma?** We chose React Native/Expo per team preference. The `llama.rn` ecosystem is more battle-tested than `react-native-litert-lm` (which has first-class Gemma support but is maintained by a single developer with 5 GitHub stars).
- **Why llama.rn over react-native-litert-lm?** Stability > convenience. llama.rn uses GGUF format with a large community. We only send text to Gemma (not images), so we don't need litert-lm's multimodal support. Can revisit if litert-lm matures.
- **Why not cloud translation?** The whole point is offline-first. Gemma runs fully on-device. No API keys, no costs, no network dependency after model download.
- **Why SQLite cache for translations?** Gemma inference costs ~1-3 seconds. Once a word is translated, it should be instant forever. SQLite gives us persistent, queryable cache that survives app restarts.
- **Future features:** Flashcard review mode, spaced repetition, share vocabulary lists, custom ML model for better labels, cloud sync of library.
